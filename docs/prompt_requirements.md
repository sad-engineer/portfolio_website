Этот документ фиксирует обязательные требования к архитектуре, структуре и качеству кода проекта.  
Требования используются как базовый контекст для формулирования промтов к ИИ и задач разработчику.

## 1) Базовая архитектурная модель (слои)

В проекте используется многослойная модель:

1. **Слой контента (Data Layer)**  
   Источник текстов и контентных сущностей: JSON-файлы в `apps/portfolio/content`.

2. **Слой шаблонов (Template Layer)**  
   Разметка страниц и компонентов: HTML/Jinja2 в `apps/portfolio/src/portfolio/templates`.

3. **Слой стилей (Style Layer)**  
   Визуальное оформление: CSS в `apps/portfolio/src/portfolio/static/css`.

4. **Слой представления (Presentation Layer)**  
   Пространственная компоновка интерактивных блоков (координаты, размеры, раскладка по id): JSON в `apps/portfolio/src/portfolio/layout/`, загрузка через `portfolio/presentation.py`. Не содержит пользовательских текстов и не дублирует поля контента (`title`, `topic` и т.п.).

5. **Слой анимации (Animation Layer)**  
   Параметры движения и таймингов: глобальный блок `animation` и покомпонентные коэффициенты (например, `growSpeed`) в layout-JSON; исполнение — в `static/js/` (например, `profession-portfolio-bubbles.js`). CSS описывает статичную оболочку арены и подписей, но не физику/таймлайн анимации.

Дополнительно:
- **Слой приложения (Application Layer)**: Python/FastAPI (`routers`, `dependencies`, `services`) отвечает за маршрутизацию, загрузку/подготовку данных, слияние контента с layout/анимацией, i18n, валидацию и безопасность.
- `routers/` — только маршруты и вызов сервисов; бизнес-логика и сборка view-model — в `services/` (`content.py`, `profession.py`, `portfolio.py`, `page_context.py`, `html_sanitize.py`).

## 2) Принципы разделения ответственности

### 2.1 Контентный слой (JSON)

Обязательно:
- Все пользовательские тексты, подписи, заголовки, CTA, пункты меню, тексты форм, legal-блоки хранятся в JSON.
- Контент хранится в структурированном виде (объекты и массивы однотипных объектов), а не в виде "магических" индексов. Если порядок важен, использовать массив объектов с явными полями (id, title, description), без доступа по магическим индексам в шаблонах.
- Для многоязычности используются единые ключи с локализациями, без привязки шаблонов к конкретному языку.
- В файлах контента придерживаться строгой структуры:
      -- Блок контента в основной локали
      -- Блок "i18n" с локалями:
            --- Блок локали "en" (пока только один)

Запрещено:
- Хардкод пользовательских текстов в HTML.
- Зависимость шаблонов от фиксированных индексов (`directions[0]`, `directions[1]` и т.п.). Если порядок является бизнес-требованием - применить именованные списки.

### 2.2 Шаблонный слой (HTML/Jinja2)

Обязательно:
- Шаблоны отвечают только за структуру и отображение.
- Шаблоны используют готовые данные из контекста (`content`, `ui`, `view_model`) без сложных вычислений.
- Повторяемые блоки выносятся в переиспользуемые partial-шаблоны.
- В шаблоне запрещено использовать жестко зашитые пути к ресурсам (например, к фонам, иконкам, изображениям). Источник ресурса должен приходить из слоя контента или подготовленного `view_model` в Application Layer.
- В шаблоне допускается только привязка ресурса к конкретному блоку/компоненту (например, через CSS-переменную, `data-*` атрибут или inline custom property), без самостоятельного выбора/вычисления пути.
- Фоновые изображения секций страниц профессий (языки, типы документов и т.п.) задаются путём к файлу в `main.json` → `assets` (например `languagesBackgroundImage`, `documentTypesSectionBackgroundImage`) и подставляются в `base.html` на `body` как `--languages-bg-image`, `--document-types-bg-image`; в CSS остаётся только `var(--…)` и описание отображения.

Запрещено:
- Сложная логика трансформации данных в шаблоне (определение типа контента, вычисление служебных флагов, бизнес-ветвления).
- Жесткая привязка к языку (`.ru`, `.en`) в разметке.

### 2.3 Стилевой слой (CSS)

Обязательно:
- Стили разделяются по файлам: базовые, layout, компоненты, страницы, медиа-правила.
- Используются CSS-переменные для токенов темы (цвета, отступы, размеры, типографика).
- Именование классов единообразное (например, BEM или согласованный project-style).
- Переменные компонента/страницы объявляются в области действия соответствующего селектора компонента/страницы (например, `.page-content.page-profession .profession-content-container > .profession-work-experience`), а не глобально в `:root`, если это не токены общей темы.
- Токены в начале CSS-файла компонента должны влиять только на целевой компонент этого файла и не должны изменять соседние секции страницы.
- Каждый CSS-блок обязан иметь краткий комментарий о назначении блока (где используется и за что отвечает).
- Каждое CSS-свойство внутри блока должно сопровождаться комментариями с объяснением, что именно оно меняет визуально/поведенчески.
- Комментарии должны писать так, чтобы можно было безопасно менять значения без обращения к внешней документации.
- Для настраиваемых значений указывать рекомендуемые диапазоны или безопасные альтернативы (если применимо).
- В каждом файле выделить в блок переменные для этого файла (т.е. если файл для компонента в этом блоке должны быть глобальны переменные для этого компонента), В переменные включать ссылки на глобальные цвета 
- В каждом CSS-блоке файла при указании цвета ссылаться на переменную из блока переменных файла (который ссылается на глобальную палитру)
- Все базовые цвета палитры задавать в формате HEX (`#RRGGBB` или `#RGB`).
- Не использовать прямые `rgb(...)`, `rgba(...)`, `hsl(...)`, `hsla(...)` для задания базового цвета; прозрачность настраивать отдельно (через `opacity`, альфа-канал в производной переменной, overlay-переменные и т.д.).
- В CSS запрещено использовать жестко зашитые пути к контентным ресурсам (`url(...)` с прямым путем к файлу), если ресурс должен быть управляемым из контента.
- CSS должен описывать только отображение ресурса (позиция, размер, повтор, фильтры и т.д.), а сам ресурс передается из шаблона/контента через переменные или атрибуты.

#### 2.3.1 Глобальная палитра цветов

Обязательно:
- Выделить единый блок глобальных цветовых переменных и расположить его в очевидном месте (например, в начале `base.css`).
- Глобальная палитра должна быть единственным источником базовых цветов проекта.
- Переменные компонентов и страниц должны ссылаться на глобальную палитру, а не содержать собственные базовые цвета.
- Для полупрозрачных состояний использовать связку "базовый цвет из глобальной палитры + отдельная переменная прозрачности".
- Разрешено использовать 8-значные HEX-значения (`#RRGGBBAA`) только как производные от глобальной палитры/overlay-токенов; базовые цвета при этом остаются в палитре.

Запрещено:
- Дублировать один и тот же базовый цвет в нескольких файлах разными литералами.
- Хранить базовые цвета только в component/page файлах без ссылки на глобальную палитру.

Запрещено:
- Монолитный CSS-файл без модульного деления при росте проекта.
- Дублирование блоков стилей и конфликтующие определения переменных/классов.
- Добавлять новые CSS-блоки/свойства без поясняющих комментариев.

### 2.4 Слой представления (layout JSON)

Обязательно:
- Файлы раскладки хранить в `apps/portfolio/src/portfolio/layout/` (например, `developer_portfolio_bubbles.json`).
- Загрузка layout — через функции в `portfolio/presentation.py` с кэшированием (`@lru_cache`); шаблоны не читают layout-файлы напрямую.
- Ключи в `bubbles` (или аналогичных секциях) совпадают с `id` проекта из `portfolio_projects.json`.
- Для каждого элемента раскладки задавать геометрию: `x`, `y`, `diameter` (координаты центра и диаметр в единицах симуляции JS).
- Application Layer обогащает контентные записи полями view-model (`bubbleX`, `bubbleY`, `bubbleDiameter`, …) перед передачей в шаблон/JS.

Запрещено:
- Хранить в layout-JSON тексты интерфейса, ссылки на ассеты и бизнес-описания проектов.
- Дублировать в layout поля, которые уже есть в контентном JSON (`title`, `topic`, `assetFolder`).

### 2.5 Слой анимации (layout + JS)

Обязательно:
- Глобальные параметры анимации — в секции `animation` того же layout-файла, что и раскладка (например, `settleDurationMs`, `growDurationMs`, `spawnFactor`, `gravity`, `repulsion`, `padding`, `damping`, `approachStrength`).
- Покомпонентные параметры движения (например, `growSpeed` на пузырёк, диапазон `0.01`–`1.0`) — в записи элемента раскладки; Application Layer нормализует и передаёт в view-model (`bubbleGrowSpeed`).
- Шаблон передаёт в JS единый конфиг (`window.__PORTFOLIO_BUBBLES__` или аналог): `items` из обогащённого контекста + `animation` из layout.
- Логика кадра, физика столкновений, easing и отрисовка SVG выполняются только в JS; Python не считает промежуточные кадры.
- Учитывать `prefers-reduced-motion`: при включённой настройке анимация должна деградировать до финального состояния без длительной физики.

Запрещено:
- Задавать тайминги и коэффициенты физики в CSS (`animation-duration` для пузырьковой физики) или хардкодить их в шаблонах.
- Дублировать одни и те же параметры анимации в JS и layout без единого источника (layout — источник для настраиваемых величин; в JS допустимы только технические константы рендера, не подлежащие правке контент-менеджером).
- Менять `viewBox`/масштаб SVG вместо координат раскладки для эффекта «расползания» кластера (движение элементов — через layout-координаты и физику, не через zoom холста).

Рекомендации:
- После правки layout-JSON перезапускать dev-сервер (кэш `@lru_cache` в `presentation.py`).
- Новый пузырёк developer-портфолио: запись в `portfolio_projects.json` + папка ассетов + запись в `developer_portfolio_bubbles.json` с `x`, `y`, `diameter`, `growSpeed`.

## 3) Требования по исключению хардкода

Под хардкодом в рамках проекта понимается:
- Тексты интерфейса в HTML/Python, которые должны быть в JSON/i18n.
- "Зашитые" URL, лейблы, названия разделов, если они относятся к редактируемому контенту.
- Жестко заданные значения, которые уже имеют централизованный источник (настройки, контент, токены стилей).

Допустимый хардкод:
- Технические константы инфраструктурного уровня (например, имена внутренних ключей, системные пути), если они не пользовательский контент и не дублируются.

## 4) Требования к i18n

Обязательно:
- Язык выбирается централизованно (query-параметр `lang`, cookie или настройки) и передается в контекст шаблонов.
- Все страницы используют текущую локаль, а не фиксированный язык.
- Есть fallback-локаль (например, `ru` или `en`) при отсутствии перевода.
- Текущая локаль определяется в Application Layer и передается в шаблон как current_lang.

Запрещено:
- Переключатель языка без реального влияния на контент.
- Смешение локалей в одной странице при штатном сценарии.

## 5) Требования к структуре файлов

Минимальная целевая структура CSS:

- `static/css/base.css` — reset, типографика, базовые переменные.
- `static/css/components/*.css` — независимые UI-компоненты.
- `static/css/pages/*.css` — страничные переопределения.
- `static/css/portfolio-layout.css` — точка сборки/подключения (импорты по выбранной стратегии).

Принятое в проекте правило размещения CSS-параметров:
- Глобальные theme-токены (цвета, базовые оверлеи, общие системные значения) — в `base.css` внутри `:root`.
- Page-токены — в `pages/<page>.css` внутри селектора страницы (например, `.page-content.page-profession`).
- Component-токены — в `components/<component>.css` внутри селектора компонента.
- Для узкоспециализированных блоков (например, блока опыта работы) токены объявляются на селекторе этого блока и применяются только к нему.

Шаблоны:
- `templates/base.html` — каркас страницы.
- `templates/partials/*.html` — переиспользуемые части (header, footer, nav, feedback, cards).
- `templates/<page>.html` — страницы, собираемые из partial-блоков.

Контент:
- Один JSON на доменную область/страницу.
- Общие тексты (навигация, footer, формы) вынесены в общий файл/секцию.

### 5.1 Стандарт хранения ассетов портфолио

Единый стандарт для **всех** проектов портфолио (конструктор, technologist, planner, developer).

Корневая папка: `apps/portfolio/src/portfolio/static/assets/projects/`.

#### Папка проекта

Обязательно:
- Имя каталога задаётся полем `assetFolder` в `portfolio_projects.json` и должно **точно совпадать** с папкой на диске, включая пробелы и кириллицу.
- Формат имени: `<id>. <название проекта>` (например, `001. МВГУ - мобильная водогрейная установка`, `301. Расчет режимов резания`).
- Подпапки по типу контента (создавать по мере необходимости, не все обязательны для каждого проекта):
  - `PNG/preview.png` — превью для карточки или пузырька.
  - `PNG/gallery/` — галерея (`.png`, `.jpg`, `.jpeg`, `.webp`).
  - `PNG/render/` — рендеры из графических программ (например, Blender).
  - `JPG/photos/` — фото изделия «в металле».
  - `JPG/render/` — рендеры из графических программ (например, Blender).
  - `PDF/` — документы для просмотра (`.pdf`); Application Layer берёт первый PDF по сортировке имени.
  - `STP/` — исходные 3D-модели (`.stp`).
  - `GLB/` — 3D для веб-просмотра (`.glb`).
  - `MP4/` — видео (`.mp4`, h264).
  - `DOCS/` — описания и вспомогательные файлы.
  - `A3D_M3D/` — 3D-модели КОМПАС.
  - `BLD/` — 3D-модели Blender.
  - `CDW/` — чертежи КОМПАС.
  - `STL/` — 3D-модели (`.stl`).

#### Запись в `portfolio_projects.json`

Обязательно:
- `id` — стабильный идентификатор; используется как ключ в layout и фильтрах профессий.
- `assetFolder` — обязателен для каждого проекта; совпадает с именем папки на диске.
- `assetProfile` — способ отображения на сайте (`standard`, `technologist-drawing`, `planner-drawing`, `developer-bubble` и т.д.).
- `preview` — при использовании нестандартного превью указывать путь к файлу **относительно папки проекта** (например, `PNG/preview.png`); по умолчанию Application Layer ожидает `PNG/preview.png`.
- При переименовании папки синхронно обновлять `assetFolder` в JSON; расхождение имён приводит к скрытию проекта (фильтр по существованию каталога).

#### Интеграция с приложением

Обязательно:
- Шаблоны получают пути к ассетам только из подготовленного контекста (`previewPath`, `documentPath`, …), без хардкода путей в HTML/CSS.
- При добавлении нового типа файлов — новая подпапка в схеме и при необходимости логика обогащения в Application Layer, без ломки существующих проектов.

Дополнительно для профиля `developer-bubble`:
- Поле `topic` (группа цвета в JS).
- Запись в `layout/developer_portfolio_bubbles.json` (`x`, `y`, `diameter`, `growSpeed`).
- Проект не попадает на страницу, если нет папки `assetFolder` или записи в layout.

Рекомендации:
- Для серий файлов в галереях — фиксированная нумерация (`01.jpg`, `02.jpg`, …).
- Для web-предпросмотра: `.mp4` (video), `.jpg`/`.webp` (изображения); исходники `.stp` хранить в `STP/` отдельно от веб-версий в `GLB/`.
- В перспективе — просмотр 3D-файлов Blender в браузере (демонстрация в трёхмерном пространстве).

## 6) Требования по безопасности

Обязательно:
- Любой HTML из контента рендерится безопасно: `|safe` допускается только для доверенного, заранее санитизированного контента.
- safe запрещен по умолчанию; исключение — только после явной серверной санации (`services/html_sanitize.py`: `sanitize_legal_html`, `sanitize_inline_html`, `linkify_phrase`).
- Внешние ссылки открываются с `rel="noopener noreferrer"` при `target="_blank"`.
- Контентные поля, попадающие в атрибуты, не должны обходить экранирование.

## 7) Требования к качеству и поддерживаемости

Обязательно:
- Минимизировать дублирование в шаблонах и CSS.
- Все новые ключи контента документировать в JSON-структуре.
- Для каждой архитектурной правки обновлять документацию в `docs`.
- Изменения не должны ломать текущие URL-роуты без явной миграции/редиректа.

## 8) Критерии приемки задачи (Definition of Done)

Задача считается выполненной, если:
- Слои не нарушены: контент в JSON, раскладка в `layout/`, анимация в layout+JS, структура в HTML, стили в CSS.
- В шаблонах отсутствует новый хардкод пользовательских текстов.
- Локализация действительно переключается и влияет на вывод.
- CSS-изменения внесены в релевантные модульные файлы, без хаотичного роста одного файла.
- Документация в `docs` актуализирована.

## 9) Шаблон требований для промтов

Использовать как обязательный блок в промтах:

1. Соблюдай архитектуру слоёв: JSON (контент), `layout/` (раскладка), layout+JS (анимация), HTML/Jinja2 (структура), CSS (стили).
2. Не добавляй хардкод пользовательских текстов в шаблоны и Python.
3. Выноси повторяемые элементы в partial-шаблоны и компонентные CSS-файлы.
4. Реализуй/сохраняй рабочую i18n-модель с централизованным выбором языка и fallback.
5. Не добавляй сложную бизнес-логику в шаблоны: подготавливай данные в Python.
6. Учитывай безопасность рендера контента (осторожно с `safe`).
7. Комментируй CSS-политику подробно: комментарий к каждому блоку и каждому свойству.
8. После изменений обнови документацию и кратко опиши, как соблюдены требования.

## 9.1 Новые UI-ключи (структура JSON)

При добавлении интерфейсных подсказок обязательно хранить тексты в `basic_content.json` в секции `ui`:

- `ui.techTileClickHint` — текст подсказки о кликабельности иконок технологий.
- Для локализаций значение ключа дублируется в `i18n.<locale>.ui.techTileClickHint`.

Использование в шаблонах/скриптах:
- Шаблон передает значение в JS-конфиг (`window.__...__`) или `data-*` атрибут.
- CSS не должен содержать хардкод пользовательского текста в `content: "..."`; текст берется через `attr(...)` из атрибута, заполненного из i18n-данных.

## 10) Типовые требования к оформлению SVG-иконок

Применяется ко всем SVG-иконкам проекта (с текстом и без текста).

### 10.1 Общие требования для любых SVG-иконок

Обязательно:
- Использовать единый формат файла: корневой `<svg>` с `xmlns`, явным `viewBox` и согласованными `width/height` для однотипных иконок.
- Для иконок одного класса (например, плитки `60x60`) держать одинаковую геометрию холста (`viewBox`, соотношение сторон, базовые отступы).
- Использовать единый базовый цвет элементов (по текущей теме проекта — белый `#ffffff`), если иконка не требует осознанной многоцветности.
- Поддерживать прозрачный фон иконки: не добавлять фоновые заливки, если они не часть смысла пиктограммы.
- Сохранять структуру SVG читаемой: один тег на строку, атрибуты на отдельных строках, корректные отступы.
- Использовать только нужные атрибуты; удалять пустые/служебные элементы после экспорта из редакторов.

### 10.2 Правила построения тегов и отступов

Обязательно:
- Каждый открывающий и закрывающий тег писать с новой строки.
- Закрывающий символ `>` писать в конце текущей строки.
- Закрывающий символ `/>` писать с новой строки.
- Использовать 2 пробела на каждый уровень вложенности.
- Для многострочных тегов (например, `<svg>`, `<path>`, `<text>`, `<rect>`) каждый атрибут писать на новой строке с дополнительным уровнем отступа.
- В блоках описания геометрии (в первую очередь атрибут `d`) ограничивать длину строки: не более `120` символов.
- Если геометрия не помещается в 120 символов, переносить ее по смысловым сегментам команд (`M`, `L`, `C`, `Q`, `A`, `Z`) без потери читаемости.
- Внутренний текстовый контент (`<text>...</text>`) размещать на отдельной строке с отступом +2 пробела относительно тега `<text>`.
- Пустые строки использовать только между логическими группами элементов (например, между фоном и основной графикой), не чаще одной подряд.
- Для монохромных иконок, цвет основной геометрии задавать явно: `fill="#ffffff"` по возможности в атрибутах корневого `<svg>` (если в файле много геометрии с одинаковым цветом, перенести цвет в корневой `<svg>` и поменять на "#ffffff").

Рекомендуемый порядок секций внутри SVG:
1. Корневой `<svg>` и его атрибуты.
2. `<defs>` (если используется).
3. Фоновые/рамочные элементы (если есть).
4. Основная графика (контуры, фигуры, линии).
5. Текстовые элементы (если есть).

Рекомендуемый порядок атрибутов в теге:
1. Геометрия и позиционирование (`x`, `y`, `cx`, `cy`, `d`, `width`, `height`, `viewBox` и т.д.).
2. Визуальные атрибуты (`fill`, `stroke`, `stroke-width`, `opacity` и т.д.).
3. Типографика (`font-family`, `font-size`, `font-weight`, `letter-spacing`, `text-anchor` и т.д.).
4. Служебные/доступность (`id`, `aria-*`, `role`, `class`) — только если действительно нужны.

Запрещено:
- Оставлять артефакты редакторов (`<title>Layer 1</title>`, пустые `<g>`, `stroke-opacity="null"`, `fill-opacity="null"`, лишние `id` без назначения).
- Смешивать разные системы координат и единицы (`pt`, `px`) в наборе однотипных иконок без необходимости.
- Добавлять декоративные элементы, не влияющие на смысл иконки, если они ухудшают читаемость на размере плитки.
- Использовать случайные/разные толщины линий, если иконки относятся к одному визуальному набору.

Рекомендации по качеству:
- Проверять читаемость иконок на фактическом размере отображения (обычно `60x60`) и на темном фоне сайта.
- Для сложных контурных иконок упрощать геометрию, чтобы мелкие детали не «шумели» в интерфейсе.
- Для парных/серийных иконок соблюдать единую визуальную «массу» (оптический размер, толщина штриха, плотность элементов).

### 10.3 Дополнительные требования для SVG-иконок с текстом

Обязательно:
- Применить 10.1 и 10.2.
- Использовать единый блок `<text>` с атрибутами позиционирования по центру: `x="30"`, `y="30"`, `text-anchor="middle"`, `dominant-baseline="central"` для иконок `60x60`.
- Для коротких аббревиатур (2-5 символов) фиксировать `textLength="45"` и `lengthAdjust="spacingAndGlyphs"` для визуально одинаковой ширины надписей.
- Использовать единый стек шрифтов: `font-family="Segoe UI, Arial, sans-serif"`.
- Применять единые параметры начертания: `font-size="45"`, `font-weight="700"`, `letter-spacing="0.5"` (если иное не требуется по макету).
- Цвет текста задавать явно: `fill="#ffffff"`.
- Текст для кодов документов хранить в верхнем регистре (`ЭМТО`, `ЭП` и т.п.).

Запрещено:
- Использовать разные шрифты/начертания в однотипных текстовых иконках одного раздела.
- Применять `style="..."` для текста, если те же параметры можно задать атрибутами `<text>`.
- Разбивать короткую аббревиатуру на несколько `<text>`-элементов без композиционной необходимости.
