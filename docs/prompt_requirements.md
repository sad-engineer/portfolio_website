Этот документ фиксирует обязательные требования к архитектуре, структуре и качеству кода проекта.  
Требования используются как базовый контекст для формулирования промтов к ИИ и задач разработчику.

## 1) Базовая архитектурная модель (слои)

В проекте используется 3-слойная модель:

1. **Слой контента (Data Layer)**  
   Источник текстов и контентных сущностей: JSON-файлы в `apps/portfolio/content`.

2. **Слой шаблонов (Template Layer)**  
   Разметка страниц и компонентов: HTML/Jinja2 в `apps/portfolio/src/portfolio/templates`.

3. **Слой представления (Style Layer)**  
   Стили, layout, визуальные компоненты: CSS в `apps/portfolio/src/portfolio/static/css`.

Дополнительно:
- **Слой приложения (Application Layer)**: Python/FastAPI (`routers`, `dependencies`, `services`) отвечает за маршрутизацию, загрузку/подготовку данных, i18n, валидацию и безопасность.

## 2) Принципы разделения ответственности

### 2.1 Контентный слой (JSON)

Обязательно:
- Все пользовательские тексты, подписи, заголовки, CTA, пункты меню, тексты форм, legal-блоки хранятся в JSON.
- Контент хранится в структурированном виде (объекты и массивы однотипных объектов), а не в виде "магических" индексов. Если порядок важен, использовать массив объектов с явными полями (id, title, description), без доступа по магическим индексам в шаблонах.
- Для многоязычности используются единые ключи с локализациями, без привязки шаблонов к конкретному языку.
- В файлах контента придерживаться строгой структуры:
      -- Блок контента в основной локали
      -- Блок "i18n" с локалями:
            --- Блок локали "EN" (пока только один)

Запрещено:
- Хардкод пользовательских текстов в HTML.
- Зависимость шаблонов от фиксированных индексов (`directions[0]`, `directions[1]` и т.п.). Если порядок является бизнес-требованием - применить именованные списки.

### 2.2 Шаблонный слой (HTML/Jinja2)

Обязательно:
- Шаблоны отвечают только за структуру и отображение.
- Шаблоны используют готовые данные из контекста (`content`, `ui`, `view_model`) без сложных вычислений.
- Повторяемые блоки выносятся в переиспользуемые partial-шаблоны.
- В шаблоне запрещено использовать жестко зашитые пути к ресурсам (например, к фонам, иконкам, изображениям). Источник ресурса должен приходить из слоя контента или подготовленного `view_model` в Application Layer.
- В шаблоне допускается только привязка ресурса к конкретному блоку/компоненту (например, через CSS-переменную, `data-*` атрибут или inline custom property), без самостоятельного выбора/вычисления пути.
- Фоновые изображения секций страниц профессий (языки, типы документов и т.п.) задаются путём к файлу в `main.json` → `assets` (например `languagesBackgroundImage`, `documentTypesSectionBackgroundImage`) и подставляются в `base.html` на `body` как `--languages-bg-image`, `--document-types-bg-image`; в CSS остаётся только `var(--…)` и описание отображения.

Запрещено:
- Сложная логика трансформации данных в шаблоне (определение типа контента, вычисление служебных флагов, бизнес-ветвления).
- Жесткая привязка к языку (`.ru`, `.en`) в разметке.

### 2.3 Стилевой слой (CSS)

Обязательно:
- Стили разделяются по файлам: базовые, layout, компоненты, страницы, медиа-правила.
- Используются CSS-переменные для токенов темы (цвета, отступы, размеры, типографика).
- Именование классов единообразное (например, BEM или согласованный project-style).
- Каждый CSS-блок обязан иметь краткий комментарий о назначении блока (где используется и за что отвечает).
- Каждое CSS-свойство внутри блока должно сопровождаться комментариями с объяснением, что именно оно меняет визуально/поведенчески.
- Комментарии должны писать так, чтобы можно было безопасно менять значения без обращения к внешней документации.
- Для настраиваемых значений указывать рекомендуемые диапазоны или безопасные альтернативы (если применимо).
- В каждом файле выделить в блок переменные для этого файла (т.е. если файл для компонента в этом блоке должны быть глобальны переменные для этого компонента), В переменные включать ссылки на глобальные цвета 
- В каждом CSS-блоке файла при указании цвета ссылаться на переменную из блока переменных файла (который ссылается на глобальную палитру)
- Все базовые цвета палитры задавать в формате HEX (`#RRGGBB` или `#RGB`).
- Не использовать прямые `rgb(...)`, `rgba(...)`, `hsl(...)`, `hsla(...)` для задания базового цвета; прозрачность настраивать отдельно (через `opacity`, альфа-канал в производной переменной, overlay-переменные и т.д.).
- В CSS запрещено использовать жестко зашитые пути к контентным ресурсам (`url(...)` с прямым путем к файлу), если ресурс должен быть управляемым из контента.
- CSS должен описывать только отображение ресурса (позиция, размер, повтор, фильтры и т.д.), а сам ресурс передается из шаблона/контента через переменные или атрибуты.

#### 2.3.1 Глобальная палитра цветов

Обязательно:
- Выделить единый блок глобальных цветовых переменных и расположить его в очевидном месте (например, в начале `base.css`).
- Глобальная палитра должна быть единственным источником базовых цветов проекта.
- Переменные компонентов и страниц должны ссылаться на глобальную палитру, а не содержать собственные базовые цвета.
- Для полупрозрачных состояний использовать связку "базовый цвет из глобальной палитры + отдельная переменная прозрачности".

Запрещено:
- Дублировать один и тот же базовый цвет в нескольких файлах разными литералами.
- Хранить базовые цвета только в component/page файлах без ссылки на глобальную палитру.

Запрещено:
- Монолитный CSS-файл без модульного деления при росте проекта.
- Дублирование блоков стилей и конфликтующие определения переменных/классов.
- Добавлять новые CSS-блоки/свойства без поясняющих комментариев.

## 3) Требования по исключению хардкода

Под хардкодом в рамках проекта понимается:
- Тексты интерфейса в HTML/Python, которые должны быть в JSON/i18n.
- "Зашитые" URL, лейблы, названия разделов, если они относятся к редактируемому контенту.
- Жестко заданные значения, которые уже имеют централизованный источник (настройки, контент, токены стилей).

Допустимый хардкод:
- Технические константы инфраструктурного уровня (например, имена внутренних ключей, системные пути), если они не пользовательский контент и не дублируются.

## 4) Требования к i18n

Обязательно:
- Язык выбирается централизованно (query-параметр `lang`, cookie или настройки) и передается в контекст шаблонов.
- Все страницы используют текущую локаль, а не фиксированный язык.
- Есть fallback-локаль (например, `ru` или `en`) при отсутствии перевода.
- Текущая локаль определяется в Application Layer и передается в шаблон как current_lang.

Запрещено:
- Переключатель языка без реального влияния на контент.
- Смешение локалей в одной странице при штатном сценарии.

## 5) Требования к структуре файлов

Минимальная целевая структура CSS:

- `static/css/base.css` — reset, типографика, базовые переменные.
- `static/css/layout.css` — сетки, контейнеры, общие layout-блоки.
- `static/css/components/*.css` — независимые UI-компоненты.
- `static/css/pages/*.css` — страничные переопределения.
- `static/css/main.css` — точка сборки/подключения (импорты по выбранной стратегии).

Шаблоны:
- `templates/base.html` — каркас страницы.
- `templates/partials/*.html` — переиспользуемые части (header, footer, nav, feedback, cards).
- `templates/<page>.html` — страницы, собираемые из partial-блоков.

Контент:
- Один JSON на доменную область/страницу.
- Общие тексты (навигация, footer, формы) вынесены в общий файл/секцию.

## 6) Требования по безопасности

Обязательно:
- Любой HTML из контента рендерится безопасно: `|safe` допускается только для доверенного, заранее санитизированного контента.
- safe запрещен по умолчанию; исключение — только после явной серверной санации.
- Внешние ссылки открываются с `rel="noopener noreferrer"` при `target="_blank"`.
- Контентные поля, попадающие в атрибуты, не должны обходить экранирование.

## 7) Требования к качеству и поддерживаемости

Обязательно:
- Минимизировать дублирование в шаблонах и CSS.
- Все новые ключи контента документировать в JSON-структуре.
- Для каждой архитектурной правки обновлять документацию в `docs`.
- Изменения не должны ломать текущие URL-роуты без явной миграции/редиректа.

## 8) Критерии приемки задачи (Definition of Done)

Задача считается выполненной, если:
- Слои не нарушены: контент в JSON, структура в HTML, стили в CSS.
- В шаблонах отсутствует новый хардкод пользовательских текстов.
- Локализация действительно переключается и влияет на вывод.
- CSS-изменения внесены в релевантные модульные файлы, без хаотичного роста одного файла.
- Документация в `docs` актуализирована.

## 9) Шаблон требований для промтов

Использовать как обязательный блок в промтах:

1. Соблюдай 3-слойную архитектуру: JSON (контент), HTML/Jinja2 (структура), CSS (стили).
2. Не добавляй хардкод пользовательских текстов в шаблоны и Python.
3. Выноси повторяемые элементы в partial-шаблоны и компонентные CSS-файлы.
4. Реализуй/сохраняй рабочую i18n-модель с централизованным выбором языка и fallback.
5. Не добавляй сложную бизнес-логику в шаблоны: подготавливай данные в Python.
6. Учитывай безопасность рендера контента (осторожно с `safe`).
7. Комментируй CSS-политику подробно: комментарий к каждому блоку и каждому свойству.
8. После изменений обнови документацию и кратко опиши, как соблюдены требования.
