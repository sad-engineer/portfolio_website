# Дорожная карта реализации формы обратной связи

Этот документ фиксирует пошаговый план внедрения рабочего функционала формы обратной связи.

## 1. Уточнение требований

- Зафиксировать обязательные поля: `phone`, минимум 1 канал связи, `consent=true`.
- Уточнить канал доставки заявок: Email / Telegram / CRM / Google Sheets.
- Определить политику хранения: только пересылка или сохранение в БД.
- Уточнить ограничения по времени приема (если требуется) и антиспам-политику.

## 2. Backend endpoint

- Добавить API-обработчик `POST /api/feedback`.
- Создать Pydantic-схему запроса:
  - `phone: str`
  - `channels: list[str]`
  - `consent: bool`
  - `page: str` (опционально)
  - `lang: str` (опционально)
- Добавить серверную валидацию:
  - телефон не пустой и валиден в базовом формате;
  - выбран хотя бы один канал связи;
  - `consent` обязательно `true`.
- Возвращать JSON-ответы со статусом и текстом ошибки/успеха.

## 3. Интеграция доставки заявок

- Реализовать отправку заявки в выбранный канал (на старте рекомендован Telegram или SMTP).
- Сформировать единый шаблон сообщения:
  - дата/время,
  - телефон,
  - каналы связи,
  - источник (страница),
  - локаль.
- Добавить обработку ошибок доставки и повторную попытку (при необходимости).

## 4. Клиентская отправка формы

- Перехватить `submit` формы через JS и отправлять данные через `fetch`.
- Добавить UI-состояния:
  - загрузка (блокировка кнопки),
  - успех (сообщение пользователю),
  - ошибка (понятный текст проблемы).
- Добавить локализованные сообщения UI в `basic_content.json` (RU/EN).

## 5. Антиспам и защита

- Добавить honeypot-поле.
- Ввести rate-limit по IP (например, N запросов за 10 минут).
- Нормализовать телефон на сервере перед отправкой.
- Логировать подозрительные сценарии.

## 6. Юридическая и комплаенс-часть

- Фиксировать факт согласия: `consent=true`, timestamp, источник.
- Проверить актуальность текста согласия и ссылок на legal-страницы.
- При необходимости добавить дополнительный флаг согласия с условиями обработки.

## 7. Тестирование и запуск

- Написать тесты API:
  - успешная отправка,
  - ошибка без согласия,
  - ошибка без каналов связи,
  - ошибка при невалидном телефоне.
- Провести ручной smoke-тест:
  - RU/EN локаль,
  - все страницы профессий и legal-страницы,
  - desktop/mobile.
- Добавить базовую диагностику в логах (успех/ошибка/причина).

## Рекомендуемый порядок внедрения

1. Endpoint + валидация.
2. Канал доставки заявок.
3. JS-отправка + UX-состояния.
4. Антиспам.
5. Тесты и финальная проверка.

## Чек-лист выполнения

- [ ] API `POST /api/feedback` реализован.
- [ ] Валидация и корректные ответы API реализованы.
- [ ] Доставка заявок подключена.
- [ ] Форма отправляется через JS и показывает статусы.
- [ ] Антиспам-защита включена.
- [ ] Тесты и ручная проверка выполнены.
- [ ] Изменения задокументированы.
